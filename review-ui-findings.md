# Cinder64 UI Design & Functionality Review

_Date: 2026-04-21. Reviewer: Claude (static review + build probe)._
_Scope: whole UI layer. Method: read every view/presentation/controller/service touching the UI, run `swift test`, attempt bundle build, plan end-to-end verification._

---

## 1. Summary

**Overall verdict: the UI design is well considered and internally consistent, but the tree does not currently work as intended — it does not compile.** Three pre-existing compile errors on `main` (HEAD = `76e6d94`) prevent `swift test`, the `.app` bundle build, the boot verification scripts, and a live launch from running. The errors are not introduced by your uncommitted changes — they are protocol drift + a macOS 15 API-removal regression that shipped in the last merge. Your uncommitted work (shutdown-timeout hardening across Swift + Rust + tests) is coherent and well structured, but it can only be validated after the pre-existing breakage is fixed.

Design-wise, the home dashboard, gameplay shell, toolbar, overlays, prompt cards, and Settings scene all follow a disciplined visual system (`ShellPalette`, rounded typography, `.thinMaterial` / `.ultraThinMaterial` surfaces, consistent `ViewThatFits` fallbacks) and presentation logic is cleanly separated into pure enums that are well tested. The policy-level tests around display mode, render-surface publication, keyboard focus, close-game coordination, launch brokering, and shell presentation are comprehensive within their slices.

What the review could _not_ cover because of the compile failure: runtime behavior, rendering, keystroke-injection smoke tests, per-metric assertions, and any observation that depends on the bundled `.app` launching.

## 2. Design observations

### Shell structure and navigation

- [ContentView.swift:24-85](Sources/Cinder64/Views/ContentView.swift) — the shell mode switch (`HomeShellView` ↔ `GameplayShellView`) is derived from a pure `ShellPresentation.mode(for: snapshot)` ([Cinder64Presentation.swift:8-12](Sources/Cinder64/Views/Cinder64Presentation.swift)) keyed on `emulationState == .stopped`. Clean, single source of truth, well covered by `Cinder64PresentationTests`.
- Prompt overlays are layered in-place via `ZStack` + `PromptOverlayContainer` with a semi-opaque backdrop ([ContentView.swift:88-119](Sources/Cinder64/Views/ContentView.swift)). This is consistent with Apple HIG for macOS blocking confirmations and avoids the awkwardness of sheet presentation on a full-bleed game window. The underlying content is correctly disabled/non-hit-testable while a prompt is shown ([ContentView.swift:158](Sources/Cinder64/Views/ContentView.swift), [ContentView.swift:671](Sources/Cinder64/Views/ContentView.swift)).

### Palette and typography

- `ShellPalette` ([ContentView.swift:121-129](Sources/Cinder64/Views/ContentView.swift)) defines a small, intentional palette: a warm accent `(0.83, 0.38, 0.20)` with two softened opacity variants, two neutral line tints, a stage shadow, and an off-black surface. Every custom color draw across the codebase goes through this enum — no drive-by `Color(red:...)` literals outside of it (checked).
- Typography is rooted in system fonts with `design: .rounded` for headline-weight text, scaling from `.caption2` → 54-pt hero in `HomeBrandHeader` ([ContentView.swift:260](Sources/Cinder64/Views/ContentView.swift)). `SettingsView` ([SettingsView.swift:12](Sources/Cinder64/Views/SettingsView.swift)) reuses the same 28-pt rounded semibold. Consistent brand voice.
- Minor copy inconsistency: `HomeBrandHeader.message` is written at second-person-imperative ("Launch a ROM…") while `HomeRecentLaunchBoard` uses third-person description ("Reopen a familiar session…"). Both read well, but if you want a single voice the recent-board copy could shift to imperative. P2.

### Responsive behavior

- `ViewThatFits(in: .horizontal)` is used consistently to cascade horizontal → vertical layouts: `HomeBrandHeader` (hero/rail), `HomeDashboardView` (recents/rail), `SessionStatusStrip` (inline/stacked), `SettingsSection` (summary/controls). This gives the window a graceful degradation path down to the 860×560 `.frame(minWidth:minHeight:)` floor set in [Cinder64App.swift:58](Sources/Cinder64/App/Cinder64App.swift).
- `HomeCanvasAccent` ([ContentView.swift:309-326](Sources/Cinder64/Views/ContentView.swift)) is `.allowsHitTesting(false)` and placed behind content. Good — it will not steal clicks from the Open ROM button.

### Empty / transitional states

- Sidebar empty state ([RecentGamesListView.swift:98-117](Sources/Cinder64/Views/RecentGamesListView.swift)) and home-board empty state ([ContentView.swift:347-358](Sources/Cinder64/Views/ContentView.swift)) both offer a primary CTA with the same "Open ROM" semantics — good.
- Surface overlay states are generated by `SurfaceOverlayPresentation.content(for:)` ([Cinder64Presentation.swift:116-144](Sources/Cinder64/Views/Cinder64Presentation.swift)) which returns `nil` for `.stopped` and `.running`, so the render surface is visually unobstructed during gameplay. The `.booting` case swaps the icon for a `ProgressView` ([RenderSurfaceView.swift:42-50](Sources/Cinder64/Views/RenderSurfaceView.swift)) — correct.

### Information density / status

- `SessionStatusStrip` currently renders a single item ("Video · <renderer> · <fps> FPS") per [Cinder64Presentation.swift:254-264](Sources/Cinder64/Views/Cinder64Presentation.swift). The strip is built for multi-item expansion (interleaves dividers between items) but only produces one. Either tighten the signature (remove the array structure) or populate it with, e.g., save-slot / audio-mute / emulation-speed once those belong in the status line. P2.
- `GameplayTitlebarTitle` ([ContentView.swift:807-820](Sources/Cinder64/Views/ContentView.swift)) couples the ROM name with a state pill — the pill's color mapping in `SessionStatePill` ([ContentView.swift:834-868](Sources/Cinder64/Views/ContentView.swift)) is well differentiated (accent / orange / red / secondary).

### Accessibility

- `.accessibilityHidden(true)` is used on `PromptBackdrop` ([ContentView.swift:98](Sources/Cinder64/Views/ContentView.swift)) — correct, the backdrop is decorative.
- Icons on toolbar items and rows are paired with titles via `Label(..., systemImage:)` — good for VoiceOver.
- Gap: no explicit accessibility labels on decorative `HomeCanvasAccent`, `SidebarBrandHeader` dot, or the `gamecontroller.fill` thumbnails in `RecentGameRow`. The default hierarchy is probably acceptable here, but an explicit `.accessibilityHidden(true)` on the decorative accents would harden it. P2.
- No `.keyboardShortcut` on the gameplay toolbar transport / reset / state / mute / display-mode menus — only the `CommandGroup` commands get shortcuts ([Cinder64App.swift:66-101](Sources/Cinder64/App/Cinder64App.swift)). This is probably fine (menu bar covers it) but worth noting. P2.

## 3. Functionality observations

### State machine correctness

- `SessionToolbarActionAvailability` ([Cinder64Presentation.swift:166-176](Sources/Cinder64/Views/Cinder64Presentation.swift)) consistently gates `canPause` on `.running`, `canResume` on `.paused`, and `canReset`/`canUseStateMenu`/`canToggleAudio` on `.running || .paused`. The toolbar disables each button using this struct ([ContentView.swift:702-760](Sources/Cinder64/Views/ContentView.swift)). Menu equivalents use the same conditions directly on `snapshot.emulationState` ([Cinder64App.swift:79-91](Sources/Cinder64/App/Cinder64App.swift)) — slightly different code path but equivalent semantics; a later refactor could share the struct.
- `CloseGameCoordinator` ([CloseGameCoordinator.swift](Sources/Cinder64/App/CloseGameCoordinator.swift)) correctly guards against double-prompts (`closePrompt == nil` check at line 73) and against closing a session with no active ROM (line 69). Save/close/cancel phase transitions are linear (`idle → saving → closing → nil`) and error-resettable. Covered by `CloseGameCoordinatorTests`.
- Protected-close save flow is symmetric: `saveProtectedCloseState` uses `SaveStateMetadataStore.protectedCloseSlot` and `hasProtectedCloseSave(for:)` is consulted before launching, producing the `ResumeProtectedSavePromptState` ([CloseGameCoordinator.swift:114-123](Sources/Cinder64/App/CloseGameCoordinator.swift)). The resume prompt is distinct from the close prompt and rendered in the Home shell, not gameplay — correct.

### ROM launch paths

All five entry points funnel through `LaunchRequestBroker.shared.enqueue(url)`:
1. File → Open ROM… (`Cmd-O`) — [Cinder64App.swift:113-125](Sources/Cinder64/App/Cinder64App.swift) via `NSOpenPanel`.
2. Home brand-header "Open ROM…" button — [ContentView.swift:270-274](Sources/Cinder64/Views/ContentView.swift).
3. Recent-launch row click — [ContentView.swift:362-363](Sources/Cinder64/Views/ContentView.swift), [RecentGamesListView.swift:31](Sources/Cinder64/Views/RecentGamesListView.swift).
4. Launch Services `open urls:` / `openFile:` / `openFiles:` — [Cinder64App.swift:272-298](Sources/Cinder64/App/Cinder64App.swift).
5. Command-line argument on launch — [LaunchRequestBroker.swift:19-24](Sources/Cinder64/App/LaunchRequestBroker.swift).

The broker standardizes URLs, deduplicates against `inFlightURL`, and serializes delivery through `isDeliveringLaunch` ([LaunchRequestBroker.swift:31-62](Sources/Cinder64/App/LaunchRequestBroker.swift)). Covered by `LaunchRequestBrokerTests`.

### Settings round-trip

- Every `SettingsView` control writes through a generic `binding(for:)` helper ([SettingsView.swift:68-79](Sources/Cinder64/Views/SettingsView.swift)) that copies `activeSettings`, mutates the key path, and calls `session.updateSettings(newSettings)`. `EmulationSession.updateSettings` persists to `PerGameSettingsStore` keyed by the active ROM identity ([EmulationSession.swift:153-165](Sources/Cinder64/Services/EmulationSession.swift)). Covered by `PerGameSettingsStoreTests`.
- `displayModeBinding` intentionally bypasses `binding(for:)` and calls `applyDisplayMode` ([SettingsView.swift:81-88](Sources/Cinder64/Views/SettingsView.swift)) because the display-mode change also needs to reshape the `NSWindow` ([Cinder64App.swift:152-160](Sources/Cinder64/App/Cinder64App.swift), [MainWindowController.swift:49-62](Sources/Cinder64/App/MainWindowController.swift)). Good separation.

### Window + scene lifecycle

- Window is single-ID `"main"` with `.restorationBehavior(.disabled)` ([Cinder64App.swift:37-64](Sources/Cinder64/App/Cinder64App.swift)) — matches `MainWindowLaunchPresentation.restoresPreviousWindowState = false` and the `window.isRestorable = false` + `window.disableSnapshotRestoration()` in [MainWindowController.swift:24-27](Sources/Cinder64/App/MainWindowController.swift).
- Reopen handling ([Cinder64App.swift:300-315](Sources/Cinder64/App/Cinder64App.swift)) delegates to `AppReopenPresentation` and calls `reopenTrackedWindowIfNeeded()` when no window is visible — this is tested in `Cinder64PresentationTests`.
- Terminate handling intercepts quit while a ROM is active ([Cinder64App.swift:317-336](Sources/Cinder64/App/Cinder64App.swift), [CloseGameCoordinator.swift:64-66](Sources/Cinder64/App/CloseGameCoordinator.swift)), re-routes through the close-game prompt, and only continues termination after `continueTerminationAfterConfirmedClose()`. Matches the `CloseGameCoordinatorTests` expectations.
- Full-screen → windowed transition goes through `didExitFullScreen` and re-applies `pendingWindowedMode` ([MainWindowController.swift:129-132](Sources/Cinder64/App/MainWindowController.swift)) — explicitly handles the known AppKit quirk where `setContentSize` during `toggleFullScreen` is lost.

### Render surface

- Publication is guarded by `RenderSurfacePublicationPolicy` ([RenderSurfaceView.swift:289-296](Sources/Cinder64/Views/RenderSurfaceView.swift)) with distinct outcomes (`clear`/`noChange`/`defer`/`publish`) tested in `RenderSurfacePublicationPolicyTests`. Live-resize is deferred, end-of-resize force-commits, `viewDidChangeBackingProperties` republishes. Solid.
- Keyboard capture is policy-driven through `RenderSurfaceKeyboardFocusPolicy.shouldRefocus` ([RenderSurfaceView.swift:357-366](Sources/Cinder64/Views/RenderSurfaceView.swift)) with coverage in `RenderSurfaceKeyboardFocusPolicyTests`.
- Repeats are filtered (`keyDown.isARepeat == false`, [RenderSurfaceView.swift:445-448](Sources/Cinder64/Views/RenderSurfaceView.swift)) so held keys don't flood the bridge.

### Gameplay toolbar

- `selectedSlot` state lives on `GameplayShellView` ([ContentView.swift:615](Sources/Cinder64/Views/ContentView.swift)) and is synced with the snapshot via `.onChange(of: snapshot.activeSaveSlot)` (line 763). This means a save/load from anywhere else in the app correctly updates the menu label. Good.
- The `Menu` content repeats the selected slot in both save and load rows ([ContentView.swift:710-724](Sources/Cinder64/Views/ContentView.swift)) — clear UX.

## 4. Test + build results

### `swift test` — **FAILED (compile error)**

Run: `swift test 2>&1 | tee /tmp/cinder64-swift-test.log` (full log in `/tmp/cinder64-swift-test.log`). Four distinct compile errors, all pre-existing on `main` HEAD (confirmed by stashing the uncommitted diff and re-running `swift build`):

**E1 — protocol conformance drift.** [Sources/Cinder64/Services/Gopher64CoreHost.swift:5](Sources/Cinder64/Services/Gopher64CoreHost.swift):
```
error: type 'Gopher64CoreHost' does not conform to protocol 'CoreHosting'
  note: protocol requires function 'enqueueKeyboardInput' with type '(EmbeddedKeyboardEvent) async throws -> ()'
  note: protocol requires function 'releaseKeyboardInput()' with type '() async throws -> ()'
```
The protocol in [CoreContracts.swift:256-257](Sources/Cinder64/Models/CoreContracts.swift) requires `enqueueKeyboardInput(_:)` and `releaseKeyboardInput()` but the host still exposes the old `setKeyboardKey(scancode:pressed:)` ([Gopher64CoreHost.swift:191](Sources/Cinder64/Services/Gopher64CoreHost.swift)).

**E2 — stale call site.** [Sources/Cinder64/Services/EmulationSession.swift:188](Sources/Cinder64/Services/EmulationSession.swift):
```
error: value of type 'any CoreHosting' has no member 'setKeyboardKey'
    try await coreHost.setKeyboardKey(scancode: event.scancode, pressed: event.isPressed)
```
Must be migrated to `enqueueKeyboardInput(event)` (and presumably there should be a `releaseKeyboardInput()` path on keyUp or when the window loses focus).

**E3 — pumpEvents signature drift.** [Sources/Cinder64/Services/EmulationSession.swift:246-249](Sources/Cinder64/Services/EmulationSession.swift):
```
error: 'async' call in a function that does not support concurrency
    if let event = coreHost.pumpEvents() {
```
The protocol declares `func pumpEvents() async -> CoreRuntimeEvent?` ([CoreContracts.swift:246](Sources/Cinder64/Models/CoreContracts.swift)) but `EmulationSession.pumpRuntimeEvents()` is synchronous. Either make the session method `async` and push that all the way up to `RenderSurfaceView.pumpRuntimeEvents`, or revert the protocol to sync. (The implementation in `Gopher64CoreHost.pumpEvents` is currently sync too, which is part of E1 in a different shape.)

**E4 — macOS 15 `CVDisplayLink` removal.** [Sources/Cinder64/Views/RuntimePumpCoordinator.swift:28](Sources/Cinder64/Views/RuntimePumpCoordinator.swift):
```
error: 'CVDisplayLink' cannot be constructed because it has no accessible initializers
    CVDisplayLinkStop(displayLink ?? .init(bitPattern: 0)!)
```
The whole `CVDisplayLink*` family is deprecated on the macOS 15 SDK and the opaque-type constructors have been removed. The `deinit` needs to guard on optional unwrap (`if let displayLink { CVDisplayLinkStop(displayLink) }`) instead of the `??` + force-init trick, and the broader API needs to migrate to `NSView.displayLink(target:selector:)` per the compiler hint.

Once E1–E4 are resolved, the UI-adjacent test files flagged in the plan (`MainWindowDisplayModeTests`, `RenderSurfacePublicationPolicyTests`, `RenderSurfaceKeyboardFocusPolicyTests`, `CloseGameCoordinatorTests`, `LaunchRequestBrokerTests`, `Cinder64PresentationTests`, `RecentGamesStoreTests`, `PerGameSettingsStoreTests`, `EmulationSessionTests`, `ScriptedKeyPlayerTests`) should be re-run as a suite; they represent the existing safety net for UI policy behavior.

### `./script/build_and_run.sh --prepare --verify` — **skipped**

Blocked by E1–E4. The bundler invokes `swift build` which will fail with the same errors. No new output collected.

## 7. Gaps and recommendations

### P0 — blockers for "works as intended"

1. **Fix protocol drift (E1 + E2).** Decide whether `setKeyboardKey(scancode:pressed:)` or `enqueueKeyboardInput/releaseKeyboardInput` is the canonical shape, then migrate the other side. The protocol in [CoreContracts.swift:256-257](Sources/Cinder64/Models/CoreContracts.swift) and the call site in [EmulationSession.swift:188](Sources/Cinder64/Services/EmulationSession.swift) must agree with the host in [Gopher64CoreHost.swift:191](Sources/Cinder64/Services/Gopher64CoreHost.swift).
2. **Reconcile `pumpEvents` async/sync (E3).** Protocol is `async`, host returns sync, caller invokes sync. Pick one. If protocol stays async, `EmulationSession.pumpRuntimeEvents` must become `async` and `RenderSurfaceHostingView.handleEventPumpTimer` needs a `Task { await ... }` wrapper or a switch to `NSView.displayLink` per E4.
3. **Migrate `RuntimePumpCoordinator` off `CVDisplayLink` (E4).** `CVDisplayLink*` C-API is deprecated on macOS 15; the `deinit` with `.init(bitPattern: 0)!` cannot compile on the current SDK. Use `NSView.displayLink(target:selector:)` and let SwiftUI own its lifetime, or fall back fully to the `Timer` path that already exists. Also resolves the `@MainActor` / `Timer` callback warning on line 122.

### P1 — improvements after unblocking

4. **Unify toolbar/menu gating on `SessionToolbarActionAvailability`.** The command-group buttons in [Cinder64App.swift:79-91](Sources/Cinder64/App/Cinder64App.swift) duplicate the `.running`/`.paused` conditions rather than going through the presentation struct. A thin helper (`SessionToolbarPresentation.actionAvailability(for:).canPause` etc.) would keep the gating in one place and extend the test coverage automatically.
5. **Hook `releaseKeyboardInput()` from the view.** Once E1/E2 are resolved, `RenderSurfaceHostingView` should call `keyboardInputChanged(.release)` (or similar) on `windowDidResignKey` / `viewWillMove(toWindow: nil)` so held keys don't stick when focus leaves the game — today it just removes the monitor ([RenderSurfaceView.swift:414-418](Sources/Cinder64/Views/RenderSurfaceView.swift)).
6. **`SessionStatusStrip` is over-engineered for one item.** If you intend to grow it, good; otherwise simplify ([Cinder64Presentation.swift:254-264](Sources/Cinder64/Views/Cinder64Presentation.swift)).

### P2 — polish

7. **Copy voice consistency** on the home dashboard (see Section 2).
8. **Decorative accessibility hinting** (`.accessibilityHidden(true)` on non-semantic accents).
9. **Consider inline `.keyboardShortcut` on toolbar items** (Space for pause/resume, `⌘M` for mute). Not required, but improves parity with the menu bar.

### Explicitly out of scope for this review

- Visual regression (no screenshot-based tests; per [CLAUDE.md](CLAUDE.md) screenshotting during a boot run destabilizes the emulator, so any screenshot tooling would have to capture outside gameplay).
- Drag-and-drop ROM automation (`application(_:open:)` is code-reviewed but not exercised).
- Accessibility audit beyond code inspection (no VoiceOver or focus-order runtime verification).
- Localization (strings are in-lined, no `NSLocalizedString`; this is a review of design/functionality, not i18n posture).
- Performance under load (`metrics.json` thresholds in the boot verifiers would have exercised this, but are currently unreachable).

## 8. Uncommitted changes note

The working tree has four modified files (4 files, +179 / −19 per `git diff --stat HEAD`). They form a coherent, well-scoped feature: **bounded shutdown so a wedged gopher64 runtime cannot stall the host process or the `applicationShouldTerminate` path.**

- [Sources/Cinder64/App/Cinder64App.swift](Sources/Cinder64/App/Cinder64App.swift) (+8/-1): `dispose` during app-quit now logs (rather than swallows with `try?`) if the shutdown path throws — useful given the new timeout error.
- [Sources/Cinder64/Services/Gopher64CoreHost.swift](Sources/Cinder64/Services/Gopher64CoreHost.swift) (+45/-11): introduces a 4-second `bridgeStopTimeout` enforced via a task group race in `performBridgeStop`; `stopRuntimeIfNeeded` and `dispose` both catch the new `Gopher64CoreHostError.shutdownTimeout` and drop the bridge handle without calling `bridge.destroy(session)` (important — the Rust side says the abandoned thread may still touch that struct).
- [ThirdParty/gopher64/cinder64_bridge/src/lib.rs](ThirdParty/gopher64/cinder64_bridge/src/lib.rs) (+72/-11): adds `join_with_timeout` using an mpsc channel, a `JoinOutcome` enum, a 3-second `SHUTDOWN_JOIN_TIMEOUT`, and `try_lock` fallback paths when a timeout left the status mutex potentially held. Returns a descriptive error when abandoning the thread.
- [ThirdParty/gopher64/cinder64_bridge/src/surface_state_tests.rs](ThirdParty/gopher64/cinder64_bridge/src/surface_state_tests.rs) (+50/0): three unit tests for `join_with_timeout` covering completed / panicked / timed-out paths, including a `Barrier`-based wedged-thread simulator that cleanly unblocks at the end of the test.

The Swift side's 4s budget intentionally sits _above_ the Rust side's 3s budget, so the Rust code gets first crack at diagnosing a stuck thread. The comments on both sides are explicit about why `destroy(session)` is skipped on timeout. The new tests on the Rust side assert timing behavior (< 1.5s elapsed for a 300ms budget). **This work is internally consistent and would be fine to commit once the pre-existing compile failures (E1–E4) are resolved.** None of the four modified files are implicated in E1–E4.
